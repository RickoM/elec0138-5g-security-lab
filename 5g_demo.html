UE — User Equipment
Simulated mobile phone. Holds USIM with subscriber credentials: SUPI (permanent identity), K (128-bit secret key), OPc (operator key). Conceals identity as SUCI using ECIES encryption before transmission over the air.
gNB — Next Generation Node B
Simulated 5G base station. Relays NAS messages between UE and AMF over the N2 interface (NGAP protocol). Handles radio resource management and forwards user plane data to UPF via GTP-U tunnels (N3).
AMF — Access and Mobility Management Function
Entry point for UE registration. Manages UE mobility, NAS security, and authentication orchestration. Calls AUSF via N12 to initiate 5G-AKA. Assigns 5G-GUTI as temporary identity after successful registration.
AUSF — Authentication Server Function
Runs 5G-AKA authentication procedure. Receives SUCI from AMF, calls UDM via N13 to get authentication vectors, stores XRES*, verifies RES* from UE, releases KSEAF on success. Rejects if RES* ≠ XRES*.
UDM/ARPF — Unified Data Management
Decrypts SUCI to recover SUPI. Runs the real Milenage algorithm (f1–f5) using K and OPc stored in MongoDB. Generates RAND, AUTN, XRES*, CK, IK and KAUSF. K never leaves UDM — this is the root of 5G security.
UPF — User Plane Function
5G data gateway. Forwards user plane packets between gNB (N3/GTP-U) and the internet (N6/SGi). Creates the ogstun TUN interface on EC2. All data traffic from the UE passes through UPF on its way to the internet.

Malware installed on SK Telecom's 5G core servers exfiltrated USIM authentication data (Ki) for an estimated 23 million subscribers. The breach began in 2022 and went undetected for three years. Novel Contribution — Transparent Decryption Proxy
The decryption proxy (mongo_proxy.py) is our original design. It sits between Open5GS and MongoDB on port 27018, intercepts BSON responses, decrypts K and OPc fields using AES-256 Fernet on the fly, and forwards plaintext to Open5GS — which never knows the credentials were encrypted. This is conceptually equivalent to AWS Nitro Enclaves, implemented as a secured software middleware running as a dedicated low-privilege systemd service. The key file is chmod 400, readable only by the mongo-proxy user.

Why the MongoDB Wire Protocol is the attack surface

📡 What is the MongoDB Wire Protocol?
MongoDB uses a custom binary protocol over TCP. Clients (including Open5GS UDR) send BSON-encoded query documents and receive BSON responses. The protocol has no built-in field-level encryption — once a client authenticates, it receives document fields in raw binary form.

When Open5GS UDR queries for a subscriber's security.k field, it arrives over the wire as a raw UTF-8 string — either plaintext hex or Fernet ciphertext, exactly as stored.
🔍 What does UDR request?
During 5G-AKA authentication, UDR queries MongoDB for the subscriber's security subdocument containing K (128-bit secret key) and OPc (operator key). UDR passes these to UDM/ARPF which runs Milenage to generate authentication vectors.

K never leaves the UDM in the 3GPP spec — but nothing in Open5GS prevents an attacker from querying MongoDB directly for the same fields UDR reads.
// Real BSON query Open5GS UDR sends:
db.subscribers.find(
  {imsi: "001010000000001"},
  {"security.k":1, "security.opc":1}
)

How mongo_proxy.py intercepts at the Wire Protocol (BSON) level
① Listen on :27018
Proxy binds to 127.0.0.1:27018. Open5GS UDR is configured to connect here instead of :27017 directly. Runs as dedicated mongo-proxy system user — not root, not ubuntu.
② Forward query verbatim
All BSON query documents from UDR are forwarded to MongoDB :27017 unchanged. The proxy is transparent on the request path — MongoDB authenticates normally with the Open5GS service account.
③ Intercept BSON response
Proxy parses the MongoDB Wire Protocol response. Detects BSON documents containing k or opc fields starting with gAAAAA (Fernet token prefix) and decrypts them in-place using AES-256 Fernet.
④ Return plaintext to UDR
Decrypted plaintext K and OPc are returned to Open5GS UDR. UDM runs Milenage normally. Open5GS never knows K was encrypted. Every decryption event is written to /var/log/mongo-proxy.log.

Rogue Network Function Registration — The 5G Core Trust Problem
In 5G, Network Functions discover each other through the NRF and communicate over the SBI using HTTP/2. Without mutual TLS, any process on the 5G core server can register as a legitimate AMF, SMF or AUSF — gaining full visibility of the internal service mesh and the ability to call any NF endpoint. This is not theoretical: our Open5GS NRF accepted a rogue AMF registration in under 100ms with zero authentication.

⚠ Real-World Incidents & Threat Context

2023 · ENISA Threat Landscape for 5G Networks
Rogue NF Registration — ENISA Top 5G Threat
The European Union Agency for Cybersecurity (ENISA) identifies rogue NF registration as one of the top threats to 5G core networks. Without mutual TLS enforcement between NFs and the NRF, an adversary with access to the SBI network segment can inject a malicious NF, use NRF discovery to map the entire core service topology, and conduct targeted SBI calls against UDM, AUSF or UDR — all while appearing as a legitimate registered network function.
🎯 Direct relevance: Our Open5GS NRF has TLS completely disabled — all configuration is commented out in nrf.yaml. The rogue registration in Sub-Tab 3 is a real attack against the live NRF, not a simulation.
2021 · AdaptiveMobile Security · Interoperability Attacks
SBI Lateral Movement in Production 5G Cores
AdaptiveMobile Security researchers demonstrated that production 5G core deployments frequently omit NRF certificate enforcement due to the operational complexity of managing PKI across all NFs. Once inside the SBI network segment — via a compromised NF, misconfigured firewall, or insider threat — an attacker can enumerate all registered NFs, their IP addresses, service endpoints and allowed caller types. This constitutes full internal reconnaissance with no authentication required.
🎯 Direct relevance: Step 2 of our attack returns the complete UDM service profile including nudm-ueau, nudm-uecm and nudm-sdm endpoints — exactly the reconnaissance an attacker needs before calling UDM.
Ongoing · 3GPP TS 33.501 §13.1 · Documented Gap
NRF OAuth2 + mTLS — Mandatory but Rarely Enforced
3GPP mandates mutual TLS between all NFs and the NRF as the primary authentication mechanism (TS 33.501 §13.1). OAuth2 token-based authorisation for SBI calls is specified in §13.3. In practice, most open-source and early commercial 5G deployments disable both — Open5GS ships with all TLS configuration commented out by default, and OAuth2 enforcement is optional. The gap between specification and deployment is the attack surface.
🎯 Mitigation demonstrated: The WAF sub-tab shows how enforcing OAuth2 token validation as an application-layer control — analogous to the Network WAF in Tab 5 — blocks the rogue NF from reaching UDM even after successful NRF registration.

What a Successful NF Hijack Achieves

① Internal Reconnaissance
NRF discovery exposes every registered NF — type, IP address, port, service endpoints, and which NF types are allowed to call each service. Complete internal network map with a single HTTP/2 GET.
② SBI Impersonation
Registered as an AMF, the rogue NF can call any endpoint that permits AMF callers — including UDM subscriber data management (nudm-sdm), UDR policy data, and AUSF authentication services.
③ Stealth — No Audit Trail
The rogue NF deregisters after completing its objective. The NRF heartbeat timer (10 seconds) also auto-expires the registration. Without NRF audit logging, the attack leaves no persistent evidence.

✓ Protected — OAuth2 Token Enforcement

How full mitigation works — mutual TLS + OAuth2 token flow
① mTLS on NRF
NRF requires a valid certificate signed by the operator CA for any NF registration. Rogue NF without a signed certificate gets HTTP 401 — cannot register at all. Configured in nrf.yaml via tls.verify_client_cacert.
② OAuth2 Token Issuance
Legitimate NFs that pass mTLS receive an OAuth2 access token from NRF scoped to specific services. Token contains NF type, instance ID, and permitted service operations — cryptographically signed by NRF.
③ Token Validation at UDM
UDM validates the Bearer token on every SBI call — verifies signature, expiry, scope, and caller NF type. A rogue NF without a valid token gets HTTP 401 from UDM regardless of NRF registration status.
④ Current State
Our Open5GS has OAuth2 partially enforced — UDM returns HTTP 400 (not 401) for missing tokens, meaning token validation is active but mTLS at NRF is not. The demo shows both the vulnerability and the partial mitigation.
